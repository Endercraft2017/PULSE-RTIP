1. PROJECT OVERVIEW
--------------------------------------------------------------------------------
PULSE-RTIP (Real-Time Incident Reporting Platform) is a disaster response and
emergency reporting system developed for the Municipal Disaster Risk Reduction
and Management Office (MDRRMO) of Morong, Rizal, Philippines. It enables
citizens to report incidents using text, images, and GPS coordinates, while
providing MDRRMO personnel with a centralized dashboard for validation,
response coordination, and mass-alert broadcasting.

Target Users:
  - Citizens of Morong, Rizal — report incidents, receive alerts
  - MDRRMO Administrators — validate reports, broadcast hazards, manage SMS

Deployment Forms:
  - Android mobile app (Capacitor, APK)
  - Web app (https://pulse.afkcube.com)
  - Optional Electron desktop build

Core Problem Addressed:
  Disaster response in Morong suffers from delayed and unverified reports, and
  from communication gaps when citizens have no internet access. PULSE-RTIP
  solves both through online incident reporting with media/GPS and an offline
  SMS channel (PULSE911 format) relayed through a TextBee Android gateway.

7. AUTHENTICATION & SECURITY MODEL
--------------------------------------------------------------------------------
- JWT (HS256) tokens with sub (user id) and role claims, 24h expiry
- Passwords hashed with bcryptjs (10 rounds); plaintext never stored
- authenticate() middleware extracts Bearer token, verifies signature,
  attaches req.user; requireAdmin() adds role gating
- Citizens see only their own reports; admins see everything
- Helmet security headers (CSP relaxed for inline scripts in SPA shell)
- CORS whitelist per environment
- Express rate limiter: 100 requests / 15 min / IP on /api
- express-validator input validation on all write endpoints
- HTTPS enforced in production via Let's Encrypt + Nginx


8. OFFLINE EMERGENCY SMS (PULSE911 PROTOCOL)
--------------------------------------------------------------------------------
A key thesis contribution: citizens without internet can still report
emergencies by SMS.

Message format (sent to TextBee gateway number):
    PULSE911|{TYPE}|{SEV}|{LAT},{LNG}|{NAME}|{PHONE}|{MESSAGE}

Codes:
    TYPE     FL=Flood, FR=Fire, TY=Typhoon, IN=Infrastructure,
             EQ=Earthquake, LS=Landslide, OT=Others
    SEV      1=low, 2=medium, 3=high
    LAT,LNG  Decimal WGS84 coordinates from device GPS
    NAME     Citizen display name
    PHONE    Citizen phone number (PH)
    MESSAGE  Short free-text description (<=100 chars)

Example:
    PULSE911|FL|3|14.5117,121.2395|Ray Lopez|09122078435|Flooded road

Processing pipeline:
    1. TextBee Android gateway receives the SMS
    2. smsPoller.js polls TextBee REST inbox every 60 seconds
    3. smsParser.js parses the PULSE911 payload
    4. A row is inserted into sms_reports (status='pending')
    5. Admin reviews in dashboard and either converts it into a formal
       incident report or dismisses it

13. NOTABLE DESIGN DECISIONS (for the thesis discussion)
--------------------------------------------------------------------------------
- Vanilla-JS frontend wrapped by Capacitor instead of React/Flutter: minimises
  dependency footprint and build complexity while keeping a single codebase
  for web, mobile, and desktop (Electron) targets.
- Dual-database abstraction (MySQL/SQLite) allows full offline development
  and field demos without a live database server.
- Stateless JWT auth simplifies horizontal scaling and mobile session reuse.
- TextBee on a low-cost Android phone serves as an SMS gateway, avoiding
  telco API fees and working end-to-end with free infrastructure.
- PULSE911 SMS protocol provides a functioning offline reporting channel in
  communities with poor mobile data coverage.
- News feed fuses the free GDACS RSS feed with NewsAPI, with relevance
  scoring to prioritise MDRRMO, Morong, and Rizal mentions.

A.3  CONCERN-BY-CONCERN CHANGES
--------------------------------------------------------------------------------

A.3.1  EDITABLE TEXTBEE SMS CREDENTIALS (ADMIN UI)
--------------------------------------------------------------------------------
Requirement: Admins can change the TextBee API key, device ID, API URL, and
gateway phone at runtime without editing the server .env or restarting the
backend.

Diagnosis: Credentials were read only from process environment variables at
startup; a change required shell access plus a service restart.

Implementation:
  * New table app_settings (migration 016) — key VARCHAR(64) primary key,
    value TEXT, updated_at, updated_by. Column `key` is back-tick quoted
    for MySQL reserved-word compatibility.
  * Model AppSetting provides get / getMany / set, with dialect-aware
    upsert (SQLite ON CONFLICT vs MySQL ON DUPLICATE KEY UPDATE).
  * services/sms/textbee.js refactored to resolve credentials via
    getEffectiveConfig() — DB overrides .env per field. An in-memory
    cache is invalidated after every admin settings save so the next
    SMS send picks up the new values without a restart.
  * Three admin-guarded routes under /api/admin/settings/sms:
        GET   — returns each field with masked apiKey + source marker
                ('db' | 'env' | 'unset')
        PUT   — upserts any subset of fields; '' clears an override
        POST /test — sends "PULSE 911 admin test ✓" to a phone
  * Admin page src/frontend/public/js/pages/admin-settings.js renders
    the creds card + Test SMS section. API key field shows only a mask
    for the current value; typing replaces it.
  * Desktop sidebar adds an "Admin" group with a Settings link. Mobile
    access via #/admin-settings.

Security:
  - GET masks apiKey (first 6 characters + bullets); PUT never echoes
    secrets back
  - Both endpoints require authenticate + requireAdmin middleware

A.3.2  ACCOUNT APPROVAL WORKFLOW (ADMIN + CITIZEN) WITH EMAIL + SMS
--------------------------------------------------------------------------------
Requirement: Every new account — citizen or admin — must be reviewed by an
MDRRMO admin before first login. On approval, the applicant receives both
SMS and email notifications; on rejection, a human-readable reason.

Diagnosis: Citizen signups were previously auto-activated (JWT on register).
Only admin-role requests queued via admin_request_status='pending'. Email
infrastructure did not exist in the codebase.

Implementation (back-end):
  * controllers/authController.js register():
      - Phone must be verified via SMS OTP in the preceding 15 minutes
      - Client-submitted role field is IGNORED; every new row uses
        role='citizen'. Server-side doc comment captures the invariant.
      - admin_request_status = 'pending_admin' if the client claimed
        admin intent, else 'pending'.
      - Response is HTTP 202 with {success:true, pending:true, kind,
        message} — no JWT.
      - All admins receive an in-app Notification with actor_user_id
        pointing at the new user.
  * controllers/authController.js login():
      - After bcrypt check, blocks admin_request_status in
        {'pending','pending_admin','rejected'} with explicit error
        codes ACCOUNT_PENDING and ACCOUNT_REJECTED.
  * controllers/adminRequestController.js approve() / reject():
      - approve(): promotes role to 'admin' only when status was
        'pending_admin'; otherwise keeps role='citizen'. Stamps
        status='approved'. Fires SMS + email + in-app notification
        (all best-effort with try/catch).
      - reject(): sets status='rejected' and fires SMS + email + in-app
        rejection notification.
  * models/User.js:
      - findPendingApprovals() returns both pending_admin + pending rows
        tagged with a `kind` discriminator.
      - findPendingAdminRequests() preserved as alias for backward compat.

Implementation (front-end):
  * utils/store.js register() handles the new 202 pending response
    shape; login() surfaces the error code to callers.
  * pages/signup.js: on pending response, renders a "Account pending
    review" screen ("Thanks! Your account is pending MDRRMO admin
    review. You'll be notified by SMS and email.") instead of auto-
    navigating to home. No token is stored.
  * pages/login.js: branches on ACCOUNT_PENDING / ACCOUNT_REJECTED /
    ACCOUNT_DELETED and shows the server's message.
  * pages/notifications.js: approval cards carry a type badge
    ("Admin Request" vs "Citizen Signup") and the ID verification
    panel (see A.3.9).

Email service (new):
  * src/backend/services/email/mailer.js — nodemailer wrapper.
      if SMTP_HOST configured → sends via the SMTP relay
      if SMTP_HOST empty      → logs "[mailer:dev] ..." to stdout
    The dev fallback is the key choice that keeps the approval flow
    runnable in any environment without a real mail account.
  * config/index.js adds SMTP_HOST, SMTP_PORT, SMTP_SECURE, SMTP_USER,
    SMTP_PASS, SMTP_FROM (default "PULSE 911 <noreply@afkcube.com>").

Future mail infrastructure: see §A.7.

A.3.3  ACCOUNT SOFT-DELETE
--------------------------------------------------------------------------------
Requirement: Any user can delete their own account; admins can delete
citizen accounts (not other admins, not themselves). Deletion must preserve
referential integrity — reports, hazards, and posts authored by a deleted
user stay visible.

Diagnosis: Frontend Delete Account button existed but the backend route

A.3.7  "VERIFICATION EXPIRED" POPUP — OTP TIMEZONE BUG FIX

A.3.10  OFFLINE SMS INCIDENT REPORTING (REPORT PATHWAY)
--------------------------------------------------------------------------------
Requirement: When the mobile APK is offline but a cell tower is available,
citizens should file a non-emergency incident report by SMS. App collects
title + description + GPS and transmits to the TextBee gateway. The
backend polls, parses, and stages for admin review.

Diagnosis: The existing offline SOS pathway (PULSE911 protocol) handled
emergencies only; the SMS format required a pre-set type/severity code
pair and was unsuitable for general incident reports. The back-end
smsParser rejected non-PULSE911 formats.

Implementation:
  * Second SMS protocol, complementing PULSE911:
        REPORT|TITLE|LAT,LNG|NAME|PHONE|MESSAGE
    Client-side constraints: title ≤ 80 chars, message ≤ 130 chars.
    User-typed '|' characters are replaced with '/' to protect the
    delimiter. Stays within one 160-char SMS segment.
  * services/sms/smsParser.js: added isReport() + parseReport() + a
    unified parse() that returns {source_type, ...fields}. isPulse911
    / parsePulse911 preserved. isSupported() helper gates the poller.
  * services/sms/smsPoller.js: gate changed from isPulse911 to
    isSupported. source_type is persisted. For REPORT messages the
    TITLE is prepended to the stored message ("Title — body") so
    admins see context without a new schema column.
  * Migration 015 adds sms_reports.source_type VARCHAR(10) NOT NULL
    DEFAULT 'sos'. Pre-existing rows stay valid.
  * models/SmsReport.js create() persists source_type (default 'sos').
  * admin-dashboard.js: each SMS report card now shows a colored
    badge differentiating SOS (red) from Report (blue). Existing
    "Convert to full report" workflow unchanged.
  * pages/report-offline.js (new): ReportOffline modal mirroring the
    SosOffline structure — Title input, Description textarea with live
    counter, GPS auto-capture (navigator.geolocation,
    enableHighAccuracy:true, 15s timeout). On submit, builds the pipe-
    delimited REPORT payload and opens sms:<gateway>?body=<encoded>
    so the user taps Send in their native SMS app. Gateway phone is
    retrieved from localStorage (cached during prior online session).
  * citizen-home.js: #sos-btn-container now contains TWO offline-only
    buttons — "SOS Emergency Report" (red) + "Report Incident (offline)"
    (green). Same offline-visibility toggle; same online/offline event
    listeners.
  * index.html registers the new script with a cache-buster.

Security: Inbound SMS NEVER auto-promote to public incidents; they land
in sms_reports and require admin action to become visible. This
preserves manual review and prevents SMS-based spam flood.

A.7  MAIL INFRASTRUCTURE — CURRENT STATE AND ROADMAP
--------------------------------------------------------------------------------
The production server hosts a receive-only Python SMTP catch-all at
/root/afkcube-mail/mail_server.py (port 25, Flask API on port 5001)
intended for OTP-capture from other services by a Chrome extension.
afkcube.com has NO outbound-mail DNS records (no MX, no SPF, no DKIM,
no DMARC). Consequently, messages from *@afkcube.com to real Gmail /
Yahoo / Outlook recipients are almost certainly rejected or spam-filtered.

Chosen interim strategy (suitable for thesis defence):
  * nodemailer is fully integrated with env-driven SMTP configuration.
  * The production server's .env leaves SMTP_HOST UNSET. The mailer's
    dev fallback logs approval / rejection email bodies to the pm2
    log. This keeps the approval contract ("SMS + email") verifiable
    without attempting unreliable delivery from an unverified domain.
  * Recommended future path (30-minute production setup):
      1. Register an account with Brevo (formerly Sendinblue) free
         tier — 300 emails/day, forever free, no credit card.
      2. Add afkcube.com as a sender domain; paste the SPF, DKIM, and
         DMARC TXT records Brevo provides into the DNS zone.
      3. Generate an SMTP key in Brevo; populate the server .env:
             SMTP_HOST=smtp-relay.brevo.com
             SMTP_PORT=587
             SMTP_USER=<brevo login>
             SMTP_PASS=<brevo SMTP key>
             SMTP_FROM=PULSE 911 <noreply@afkcube.com>
      4. pm2 restart pulse-rtip-api.
  No application code changes are required at switch-over time — the
  mailer auto-detects SMTP_HOST and swaps from dev-log to real send.

Alternative full-mailbox option for future work: Zoho Mail Lite free
tier (5 mailboxes × 5 GB, forever free) — requires replacing the
afkcube.com MX record, which would break the existing Python catch-all
unless that service is moved to a subdomain.

B.3.9  PUSH NOTIFICATIONS (U-7 + A-4)
--------------------------------------------------------------------------------
Migration 024 creates push_tokens (token VARCHAR(500) UNIQUE, user_id
nullable, platform, barangay, last_seen_at, created_at). Tokens registered
BEFORE login are unclaimed (user_id=NULL) and still receive broadcasts — the
pre-login barangay can be supplied by the client. On login, a claim() step
associates the token with the user and inherits users.barangay.

Capacitor integration:
  * @capacitor/push-notifications@8.0.3 added as a dependency.
  * @capacitor/app@8.1.0 added alongside (for U-12's hardware-back listener).
  * capacitor.config.ts exposes PushNotifications presentation options
    (badge, sound, alert).
  * src/frontend/public/js/utils/push.js handles the lifecycle:
    requestPermissions → register → listen for registration → POST
    /api/push-tokens. On login, claimAfterLogin() attaches the token.

FCM sender (src/backend/services/push/fcm.js):
  * isConfigured() returns false when FCM_SERVER_KEY is unset → send() logs
    payloads to stdout instead of dispatching. This keeps broadcasts working
    end-to-end in dev without FCM credentials.
  * Batches of 500 tokens per HTTP call (FCM legacy HTTP).
  * Each push carries { title, body, sound: 'default' | undefined,
                        priority: 'high', data }.

Hazard broadcasts now do SMS + push in parallel. Audience filter applies to
both channels.

the stack is empty. Login/signup/forgot-password routes clear the stack so
a fresh session starts clean.

U-13 — Dark mode button contrast: styles.css gained dark-mode overrides for
.btn--primary, .btn--danger, .btn--outline, .btn--ghost, .btn--link, and
.header__back (explicit #FFFFFF text on primary/danger; greened-out outline
variant).

draggable iOS-style pill. Touch + mouse supported; 80% drag threshold
triggers the `tel:` URL. Handle animates back to start if released early.

U-16 — Carrier indicator: new carrier-lookup.js utility exposes a prefix
→ carrier map (Globe / Smart / DITO / Landline / Unknown). Emergency page
phone numbers render a small pill badge alongside each number. Dark mode
brightness-adjusted automatically via .carrier-badge body.theme-dark.

broadcast-delivery notifications).
